# المشروع: منصة محاسبة بمساعدة الذكاء الاصطناعي (SaaS)

منصة ويب باشتراك شهري تساعد أصحاب الأعمال الصغيرة والمتوسطة ومكاتب المحاسبة على
المحاسبة اليومية وإصدار القوائم المالية، بمساعدة الذكاء الاصطناعي (قراءة الفواتير،
الترحيل التلقائي، الرد على أسئلة مالية، اقتراح التصنيف). الأسواق المستهدفة: مصر،
السعودية، الإمارات، الأردن، لبنان — لكل دولة نظام فاتورة إلكترونية مختلف، فالـ MVP
يبدأ بدولة واحدة فقط (راجع MVP-PRD.md).

## المرجع الكامل للخطة

راجع ملف BUSINESS-PLAN.md في جذر المشروع لأي سؤال عن الاستراتيجية، التسعير، أو
المتطلبات التنظيمية لكل دولة. لا تعيد اتخاذ قرارات استراتيجية موجودة فيه بدون سبب.

## التقنيات المستخدمة

- **الواجهة والخلفية**: Next.js (App Router) + TypeScript.
- **قاعدة البيانات**: PostgreSQL عبر Prisma ORM.
- **المصادقة**: NextAuth (أو ما يعادله) مع دعم تعدد المستأجرين (multi-tenant: كل
  منشأة/عميل = tenant مستقل بياناته معزولة بالكامل).
- **الذكاء الاصطناعي**: Anthropic Claude API (نموذج Vision لقراءة صور/PDF الفواتير،
  ونموذج نصي للشات بوت والتصنيف).
- **الاستضافة المقترحة للـ MVP**: Vercel + قاعدة بيانات مُدارة (مثل Neon/Supabase).

## مبدأ معماري أساسي (لا تخالفه)

النظام المحاسبي الداخلي **دفتر يومية مزدوج القيد (double-entry ledger)** حتى لو
الواجهة مبسطة لغير المحاسبين. كل حركة مالية = قيد محاسبي (Journal Entry) فيه سطور
دائن/مدين متوازنة. القوائم المالية (الدخل، الميزانية) تُحسب من الـ ledger مباشرة،
لا تُخزَّن كأرقام منفصلة.

## معمارية الامتثال الضريبي (Compliance Module)

لا تكتب منطق ضريبي أو فوترة إلكترونية خاص بدولة معينة داخل الكود الأساسي. كل دولة
لها وحدة امتثال منفصلة (مثل `compliance/egypt`, `compliance/saudi`) تتعامل مع:
صيغة الفاتورة، التوقيع/الختم الإلكتروني، الربط مع الجهة الضريبية، ونسبة الضريبة.
الكود الأساسي (الـ ledger والتقارير) يبقى عامًا لكل الدول.

## دور الذكاء الاصطناعي — قيد مهم

الذكاء الاصطناعي **يقترح** ولا **يُرحّل تلقائيًا** بدون تأكيد من المستخدم، خصوصًا
في: تصنيف الحسابات، استخراج بيانات الفاتورة، وأي قيد محاسبي جديد. دائمًا اعرض شاشة
مراجعة/تأكيد قبل حفظ أي قيد ناتج عن الذكاء الاصطناعي.

## نطاق العمل الحالي

ابدأ فقط بما هو موصوف في MVP-PRD.md. لا تضف ميزات من باقي مراحل الخطة (تسوية بنكية،
لوحة محاسبين متعددة العملاء، دول إضافية) قبل اكتمال واختبار MVP.

## أسلوب الكود

- TypeScript صارم (strict mode)، تسمية متغيرات بالإنجليزية، تعليقات بالعربي أو
  الإنجليزي حسب وضوح السياق.
- اختبارات وحدة أساسية لمنطق الـ ledger (لازم كل قيد يكون متوازن دائمًا).
- لا تستخدم localStorage/sessionStorage لأي بيانات مالية حساسة.
